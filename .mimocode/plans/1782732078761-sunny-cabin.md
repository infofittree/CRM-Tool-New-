# Plan: Lead Transfer (Handover) System

## Architecture Insight

`Lead.assigned_to` is the single source of truth for all ownership — tasks, dashboard, analytics, and follow-up access all derive from it. Once a lead is reassigned, everything cascades automatically. The key challenge is building a **pending/accept/decline** workflow on top of this, with notifications and audit trail.

Add after the existing `LeadTransfer` model (~line 331):

```python
class LeadHandover(Base):
    __tablename__ = "lead_handovers"
    id: int (PK, autoincrement)
    lead_id: str (String(32), NOT NULL, indexed)
    from_user: str (String(100), NOT NULL)

    responded_at: datetime | None
    responded_by: str | None (String(100))
    created_by: str (String(100), NOT NULL)
```

### 3. Backend: `api/routers/transfers.py` — New router with 5 endpoints

```
POST   /leads/{lead_id}/handover          — Create handover request
GET    /leads/{lead_id}/handovers         — History for a lead
GET    /me/handovers                      — Pending handovers for current user
POST   /handovers/{id}/accept             — Accept a handover
POST   /handovers/{id}/decline            — Decline a handover
```

**Create handover** flow:
1. Validate: lead exists, not deleted, not completed
2. Validate: from_user == current user's full_name (or Admin/Manager)
3. Validate: to_user exists, is active, is Salesperson, different from from_user
4. Validate: no existing PENDING handover for this lead
5. Create `LeadHandover` record with status=PENDING
6. Create `CrmAlert` for recipient: `{alert_type: "handover_request", lead_id, message: "Rahul wants to transfer ABC Exports to you — Reason: Product Expertise", assigned_to: to_user}`
7. Log `ActivityLog` with action="HANDOVER_REQUESTED"
8. Return the handover record

**Accept handover** flow:
1. Validate: current user == to_user
2. Validate: status == PENDING
3. Update handover status → ACCEPTED, set responded_at
4. Update `Lead.assigned_to = to_user`
5. Update all `FollowUp` rows for this lead: set `assigned_to = to_user` (so they appear in recipient's task engine)
6. Update all `EngagementEvent` rows for this lead: set `user_name` if applicable
7. Update `Inquiry.created_by` for this lead's inquiries: set to new owner
8. Create `LeadTransfer` audit record (existing model) for backwards compatibility
9. Create `CrmAlert` for sender: `{message: "Kusum accepted transfer of ABC Exports", assigned_to: from_user}`
10. Log `ActivityLog` with action="HANDOVER_ACCEPTED"
11. Return success

**Decline handover** flow:
1. Validate: current user == to_user
2. Update status → DECLINED, set responded_at
3. Create `CrmAlert` for sender: notification of decline
4. Log `ActivityLog` with action="HANDOVER_DECLINED"

## Edge Cases Handled

| Case | Handling |
|------|----------|
| Transfer to self | Backend rejects: "Cannot transfer to yourself" |
| Transfer completed lead | Backend rejects: "Cannot transfer completed leads" |
| Transfer deleted/archived lead | Backend rejects: "Lead not found" |
| Duplicate pending transfer | Backend rejects: "Transfer already pending" |
| Transfer to inactive user | Backend rejects: "Recipient is not active" |
| Transfer to non-Salesperson | Backend rejects: "Only salespersons can receive transfers" |
| Decline by wrong user | Backend rejects: "Not authorized" |
| Accept when already responded | Backend rejects: "Transfer already processed" |
