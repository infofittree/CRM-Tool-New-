# Plan: Procurement Negotiation & Re-Inquiry Workflow

## Core Principle

One customer conversation = one inquiry. Revisions extend the existing inquiry via an `inquiry_revisions` junction table. No duplicate inquiry records.

## Architecture

### New table: `inquiry_revisions`

| Field | Type | Description |
|-------|------|-------------|
| id | int PK | Auto-increment |
| inquiry_id | int FK ‚Üí inquiries.id | Parent inquiry |
| revision_number | int | Sequential (1, 2, 3...) |
| created_by | str(100) | Salesperson who requested |
| reason | str(50) | Enum: budget, competitor, discount, quantity, spec, packaging, delivery, payment, freight, other_quotation, customer_feedback |
| customer_feedback | text | Free-text customer comments |
| target_price | str(50) | Optional target price |
| quantity | str(50) | Optional quantity change |
| packaging | str(50) | Optional packaging change |
| delivery_timeline | str(50) | Optional delivery change |
| payment_terms | str(50) | Optional payment change |
| additional_requirements | text | Optional extra needs |
| status | str(20) | PENDING ‚Üí RESPONDED |
| created_at | datetime | When revision was requested |
| responded_at | datetime | When procurement replied |
| responded_by | str(100) | Who responded |

### Backend: `api/routers/inquiries.py` ‚Äî Add 3 endpoints

1. **POST /inquiries/{id}/revision** ‚Äî Create revision request
   - Validates: inquiry is in RESPONDED state, user is creator/Admin/Manager
   - Creates `InquiryRevision` row
   - Updates inquiry status ‚Üí REVISION_REQUESTED
   - Creates CrmAlert for procurement
   - Logs ActivityLog

2. **POST /inquiries/{id}/revisions/{rev_id}/respond** ‚Äî Procurement responds to revision
   - Validates: inquiry status is REVISION_REQUESTED, user is Procurement/Admin/Manager
   - Sets revision status ‚Üí RESPONDED
   - Updates inquiry response text
   - Sets inquiry status ‚Üí REVISED_RESPONSE
   - Creates CrmAlert for sales
   - Logs ActivityLog

3. **GET /inquiries/{id}/revisions** ‚Äî List all revisions for an inquiry
   - Returns chronological list of revisions with status

## Status Flow (Updated)

```
OPEN ‚Üí EOD_COMMITTED ‚Üí RESPONDED ‚Üí Sales requests revision
                                 ‚Üí REVISION_REQUESTED
                                 ‚Üí Procurement responds
                                 ‚Üí REVISED_RESPONSE
                                 ‚Üí (cycle repeats or close)
                                 ‚Üí CLOSED
```

## Verification

1. Create inquiry ‚Üí procurement responds ‚Üí Sales sees "Request Revision"
2. Sales submits revision ‚Üí procurement sees "Revision Request" badge
3. Procurement responds to revision ‚Üí Sales sees "Revised Quote"
4. Sales can request another revision ‚Üí revision #2 created
5. Procurement responds again ‚Üí cycle continues
6. Inquiry timeline shows full conversation
7. Dashboard counts update for both sales and procurement
8. All existing inquiry workflows unaffected
