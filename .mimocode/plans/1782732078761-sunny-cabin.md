# Plan: Dashboard Context Filter for Admin & Manager

## Motivation

Managers need to quickly check individual salesperson workloads from the Dashboard without switching to Analytics or logging in as that user. This must be independent of the Analytics page filter — they share no state.

## Architecture Finding

The existing `SalespersonContext` (used by both Dashboard and Analytics) is shared. Selecting a person on Dashboard also filters Analytics. The fix: Dashboard manages its own filter state locally, and hooks accept an optional override parameter.

**Backend**: All dashboard endpoints already accept `salesperson` query param. No backend changes needed. Task queue and alerts lack backend filtering — handled via client-side filter on `assigned_to`.

## Files to Modify

### 1. `web/src/hooks/useDashboard.ts` — Add optional parameter to all hooks

Each hook gets an optional `salespersonOverride?: string | null` parameter. When provided, it takes precedence over the context-based `sp()` value.

```ts
// Before:
export function useDashboardCounts() {
  const s = sp();
  ...
}

// After:
export function useDashboardCounts(salespersonOverride?: string | null) {
  const s = salespersonOverride ?? sp();
  ...
}
```

Apply to: `useDashboardCounts`, `usePipelineHealth`, `useDashboardLeads`, `useExecutiveSummary`, `useProductivity`, `useSalespersonKpi`, `useSalespersonStats`, `useRecentActivities`, `useEngagementStats`, `useTodayPriorities`.

For `useTaskQueue`: no backend param exists. Add optional override that enables client-side filtering:
```ts
export function useTaskQueue(upcomingDays = 7, maxToday = 20, salespersonFilter?: string | null) {
  const { data, ...rest } = useQuery({ ... });
  const filtered = useMemo(() => {
    if (!salespersonFilter || !data) return data;
    const byPerson = (t: Task) => t.assigned_to?.toLowerCase() === salespersonFilter.toLowerCase();
    return { ...data, today_capped: data.today_capped.filter(byPerson), overdue: data.overdue.filter(byPerson), upcoming: data.upcoming.filter(byPerson) };
  }, [data, salespersonFilter]);
  return { data: filtered, ...rest };
}
```

For `useAlerts`: same pattern — client-side filter by `assigned_to`.

### 2. `web/src/pages/Dashboard.tsx` — Add filter UI + wire to hooks

**Changes:**
- Add local state: `const [dashboardPerson, setDashboardPerson] = useState<string | null>(null)`
- Add user list fetch: `useEffect` to fetch `/users/salespersons` for Admin/Manager
- Render dropdown in dashboard header area (between hero and context banner)
- Pass `dashboardPerson` to all hook calls as override
- Show context banner when filter is active
- Reset filter on role change or unmount

**UI placement**: Between the hero banner and the action items section. A compact row with a `<select>` dropdown labeled "Viewing Dashboard For" with "Entire Team" as default.

**What changes when a person is selected:**
- Hero banner text reflects the person's data (counts from filtered queries)
- Action Items (Overdue, No Follow-up, At Risk, Pending Inquiries) all update because their data hooks receive the override
- Pipeline Health, Pipeline Summary, Inquiry Status all update (hooks receive override)
- Top Producers remains team-wide (intentional — it's a comparison widget)

**What does NOT change:**
- Analytics page filter state
- Shared SalespersonContext
- Backend endpoints
- Task engine, activity engine, notifications
